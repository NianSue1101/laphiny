# Laphiny 产品策略

## 当前产品定义

Laphiny 是一个面向多个 Hermes Agent 的本地优先协作客户端。它最强的方向不是“一个助手回答一个用户”，而是让一组拥有稳定身份的 Agent 在房间里持续协作：它们有公开人格卡片、头像、名称、角色、委托关系、房间记忆、协作仪式，以及工作和 RP 两种使用场景。

## 可能偏重或重复的功能

- 独立房间管理页适合桌面端和批量编辑，但不应该成为手机端的主要路径。手机端更应该在聊天选择页展开当前房间，直接调整名称、成员、模式、上下文和头像。
- 灵庭事件、诊断日志、房间总结、记忆胶囊、委托档案都在描述“发生了什么”。它们需要保留清晰分工：诊断只管失败和排障，灵庭只管动态沉淀，记忆只管长期决策，档案只管交接。
- 角色扮演控制应该保持可发现，但不应该压过默认聊天流程。Laphiny 的日常体验仍然要像一个快而顺手的协作房间。
- 设置页里备份、诊断、反馈都属于“数据与日志”，不应该拆成多个孤立大栏。合并后更容易理解数据流向，也更符合隐私提醒。

## 近期缺少的能力

- 首次使用时更清晰的“连接 Hermes”流程：在创建房间前先验证 endpoint、model、session 和权限。
- 更完整的手机房间编辑器：目前已覆盖名称、成员启用、模式、上下文和头像，后续可以继续补 alias、摘要者、自动委托深度。
- 可自托管的反馈闭环：用户可以上传脱敏诊断包，并从服务器拉回日志，不必 SSH 到服务器看文件。
- Agent 生成附件的稳定契约：现在已支持 `laphiny-file` 和“文件名 + 代码块”兜底，后续应该在 UI 和系统提示中更明确地告诉 Agent 推荐格式。
- 隐私状态面板：用户应该能一眼看到哪些数据只在本地、哪些会进入备份、哪些会同步、哪些会作为反馈上传。

## 差异化方向

大多数相邻产品优化的是单 Agent 聊天、提示词库、通用工作流或托管自动化。Laphiny 更适合打出“人类指挥的 Agent 小队 / Agent 社会”这条路线：

- Soul-native 身份：Agent 不是临时 prompt 槽位，而是有公开协作卡片、头像、名称、角色和长期房间关系的稳定合作者。
- 房间记忆是一等对象：决策、未解决问题、偏好、交接备注属于房间，可以指导后续回合。
- 委托是可见对话，不是隐藏自动化：Agent 可以互相请求工作、汇报状态、请求用户授权，并留下用户能读懂的审计轨迹。
- 工作和 RP 共用同一套房间原语：代码审查委员会、红队评审、复盘、长篇 RP 和 GM 模式都可以在同一产品结构里自然存在。
- 本地优先的信任边界：API Key、房间和备份由用户掌控；云同步和反馈都是明确的可选层。

## 最值得下注的独有功能

最强的下注方向是“拥有显式社会记忆的持久 Agent 房间”。Laphiny 应该让每个 Agent 像一个会反复出现的合作者，而不是一次性的补全端点。

这条路线的优势来自这些能力叠加：

- 手机端快速切房间与内联管理；
- 可见的 Agent 间委托；
- 长期房间记忆胶囊；
- 权限感知的工具请求；
- 本地优先存储、备份和导出；
- 用户可以自托管的同步与反馈服务。

如果这些能力继续收紧到一起，Laphiny 就不只是一个聊天壳，而会更像个人 Agent Studio：用户在这里维护一支会彼此协作、会记得关系、也会留下工作脉络的数字小队。
